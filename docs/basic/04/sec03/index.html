4.3 통계의 붕괴: 희소성의 폭발 (Sparsity)

전 챕터에서 빈도 확률 카운팅을 구하기 위해 분모와 분자를 카운트하여 나누던 초등학생 수준의 나누기 통계 모델은 사실 현실의 서버 환경에서는 아예 박살이 나서 쓸 수 없었던 치명적인 ‘기획 단 단계 결함 공식’이었습니다. 인터넷 구글 데이터 망에 아무리 텍스트 책이 100만 권 이상 많아도 절대 채워지지 않는 거대한 확률의 싱크홀(구멍), 데이터 희소성(Sparsity)의 덫을 수학적으로 살펴봅니다.

4.3.1 카운트 기반 언어 모델의 치명적 함정

이전 장에서 배운 조건부 확률($P$) 곱셈 공식은 칠판 위에서는 완벽해 보이지만, 철저하게 “수학적 함정” 하나를 감추고 있습니다.

\[P(\text{is} \mid \text{An adorable little boy}) = \frac{\text{Count}(\text{An adorable little boy is})}{\text{Count}(\text{An adorable little boy})}\]

이 분수 공식이 에러 없이 성립하려면 매우 당연하고도 역겨운 하나의 전제가 필요합니다. 세상 어딘가의 물리적인 서버 하드디스크에는, 우리가 물어보는 저 기다란 문맥 조합 텍스트가 통째로 저장된 Count 모수가 무조건 1개 이상 미리 세팅되어 존재해야만 합니다.

4.3.2 내가 쓴 소설 문장이 지구상에 단 하나도 없다면?

사용자가 창의성을 발휘하여 아주 특이한 문학적인 수식어를 연거푸 붙여서 챗봇에게 아래와 같이 타이핑을 쳤다고 가정해 봅시다.

“지옥의 유황불에서 디스코 춤을 추는 파란색 솜사탕이 ( ? )”

이 문장은 문법적으로나 국어학적으로는 아무런 오류가 없는 완벽한 문장입니다. 하지만 구글이나 네이버 서버의 거대한 텍스트 데이터베이스(Corpus) 전체를 다 샅샅이 뒤져봐도, 혹은 100년 치 조선일보 신문 기사와 디시인사이드 댓글들을 다 스캔해 봐도, 저 문맥 토큰 배열 덩어리가 완벽히 일치하는 데이터는 단 한 줄도 우주에 존재하지 않습니다! 아무도 저런 기괴한 문맥의 덩어리를 인터넷에 카운팅 되도록 쓴 적이 없기 때문입니다.

4.3.3 분모가 0이 되는 참사 (Zero Division Error)

이때 서버의 데이터베이스 조회 결과(Count) 함수는 자비 없이 기계적인 숫자 0을 반환합니다. 이 0이 분모에 깔리는 순간 지옥이 열립니다.

\[P(\text{is} \mid \text{지옥의 솜사탕}) = \frac{\text{Count}(\text{지옥의 솜사탕 is})}{\mathbf{0}}\]

[!CAUTION]
희소 문제 (Sparsity Problem)의 종말
전 세계 어떤 구글의 슈퍼컴퓨터라도 현대 수학에서 분모(밑바닥)가 0이 되는 연산은 물리적/논리적으로 처리할 수 없습니다! (Division by Zero) $\to$ 파이썬(Python) 엔진 프로그램은 이 수식을 만나는 즉시 치명적인 ZeroDivisionError 예외(Exception) 창을 띄우며 번역기 챗봇 서버 자체를 강제로 블루스크린 띄우고 셧다운(Shut-down) 시켜버립니다.

기계가 매끄럽게 번역을 하려면 이론적으로 ‘무한대’에 가까운 모든 텍스트 문맥 조합 쌍 데이터가 필요한데, 우주상의 어떤 구글 데이터 서버의 용량이라 할지라도 인간이 혓바닥으로 지어낼 수 있는 수백억 갈래의 단어 순서 조합(Sequence)을 다 엑셀로 저장해 둘 수는 없습니다. 이렇게 인간 언어의 무한한 공간 대비, 실제로 우리가 글로 적어서 데이터로 존재하는 표본은 극히 일부라서 대부분의 메모리 매트릭스가 텅 비어 있는 끔찍한 현상을 희소성(Sparsity) 문제라고 부릅니다.

4.3.4 기하급수 멸망의 법칙 (긴 문장에 대한 통계 불가능의 영역)

인간은 문장을 말할 때 단어 2개 수준으로 끝내지 않습니다. 문장이 기하급수적으로 서술어를 붙이며 길어질수록, 그 긴 문장 덩어리 시퀀스가 인터넷 서버에 통째로 우연히 똑같이 한 토시도 안 틀리고 존재할 확률은 로또 1등에 맞는 것보다도 더 극악하게 땅바닥으로 쳐박힙니다.

  • Count("소년이") $\to$ 너무나 평범한 2-gram 명사구입니다. 구글에 검색하면 100만 건 카운트가 거뜬히 나옵니다. (안전함)
  • Count("착한 소년이") $\to$ 수식어가 붙기 시작하니 검색 결과가 5천 건으로 팍 기가 죽습니다.
  • Count("어제 스코틀랜드에서 전학 온 매우 착한 안경 쓴 소년이") $\to$ 단 한 건도 없습니다!
  • 어순 파괴: 심지어 저 문장에서 “안경 쓴”과 “착한” 의 어순을 살짝만 뒤바꿔도 컴퓨터 입장에서는 아예 완전히 새로운 문자열 해시(Hash)로 취급하여 또 다시 카운트 값이 0 (Sparsity 폭발 에러!) 이 뜹니다.

4.3.5 진퇴양난에 빠진 통계 학자들: 꼼수의 시작

조건부 확률의 연쇄 법칙(도미노 체인) 수식으로 긴 문장을 끝까지 완벽하게 이어가고 싶었던 옛날 언어모델(SLM) 통계 학자들은 이 카운트 제로($0$) 장벽 앞에서 절망했습니다. 문장이 고작 3단어, 4단어 이상 길어지기만 하면 구글 데이터베이스에 검색 결과 0이 떠서 연쇄 확률 곱셈 파동이 중간에 강제로 셧다운 파괴되어 버렸기 때문입니다.

이를 타개하기 위해, 학자들은 우아한 체인 수식의 자존심을 굽히고 치욕적인 선언을 합니다. “아, 전체 어제 무슨 일이 있었는지 모든 문맥(Context History)을 완벽히 다 고려해서 확률을 구하는 건 서버 용량상 수학적으로 수학적으로 에바 참치다. 그냥 어거지로 편법을 써서 옛날 기억은 강박적으로 다 지워버리고 눈 가리고 아웅하자!” 라며 잔머리를 굴린 대안을 긴급히 내놓게 됩니다.

그 타협의 산물이 바로 다음 장에서 소개될, 현재의 모든 토큰 생성 알고리즘의 뼈대가 되는 N-gram 마르코프 체인(Markov Chain) 모델입니다.
